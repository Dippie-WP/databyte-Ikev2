# Portal Recovery Incident — 2026-06-27

## What happened

A code review on 2026-06-27 discovered that the `main` branch was in a
broken state because of two destructive commits shipped the previous day:

1. **`e4a4673`** — *"v1.4.0: nft METER quota + swanctl EAP regex fix +
   repo alignment (replay)"*. Deleted 117 files / 15,449 lines, including
   the entire test suite, deploy-safety script, and bug-history tracker.
   The commit message claimed *"The resulting tree is identical to
   f4ea70c"* — it was not.
2. **`5b2c2ab`** — *"docs(architecture): v2.0 — full system rewrite
   verified live on VPS"*. Stripped the operator-authentication functions
   from `portal_auth.py`, leaving the portal unable to import.

Net effect: any clean clone + `git pull` would crash the portal at
import time. The live VPS was unaffected only because it was running
a pre-replay file snapshot.

## What we did to fix it

| Step | Action | Outcome |
|---|---|---|
| 1 | Saved backup branch `backup-broken-v1.9.1-pre-reset` | Pre-recovery state preserved |
| 2 | Reset local `main` to commit `01a475f` (known-good pre-replay) | Working code restored |
| 3 | Disabled branch protection temporarily | Allowed force-push |
| 4 | Force-pushed `main` to `01a475f` on origin | Public repo state corrected |
| 5 | Deleted broken tags: `v1.4.0`, `v1.8.0`, `v1.9.0-sse` | No more confusing "latest release" |
| 6 | Cut tag `v1.7.0-recovered` at the working commit | Clean baseline established |
| 7 | Deleted `fix/post-replay-recovery` branch | Refs cleaned |
| 8 | Deleted `main-zun-v1.4.0` historical branch | Refs cleaned |
| 9 | Deleted 5 stale closed-PR refs (`refs/pull/1/head` … `5/head`) | Refs cleaned |
| 10 | Re-enabled branch protection | Safety restored |

## Final state

- `main` → `01a475f` (working, tagged `v1.7.0-recovered`)
- No portal tags other than `v1.7.0-recovered` remain
- No historical branches remain
- `.env.*` is gitignored (so secrets can't be accidentally committed)
- No secrets are present in any git history (verified)

## Lessons (don't repeat this)

- **Never** ship a "replay" or "rewrite" commit that wholesale replaces
  the file tree. The commit message claimed parity but the actual
  file diff was 117 files deleted. Always diff the trees before and
  after such a commit.
- **Always** add every entry-point module to the deploy SHA-verify
  list. The previous validator only checked `app.py`; that's how
  the desync between git HEAD and the deployed `portal_auth.py`
  went unnoticed.
- **Always** test from a fresh `git clone` before assuming the
  current `main` is runnable.
- **Always** use general `.gitignore` patterns (`.env.*`) rather than
  enumerating specific filenames. The live `.env.xneelo` was
  `git add`-able for months because the `.gitignore` only listed
  `.env` and `.env.local`.
