# vpn-prod-01 firewall (host iptables)

OS-level firewall for vpn-prod-01 (154.65.110.44). Complements the Xneelo
cloud security group (which is the primary Cloudflare-IP allowlist filter).

## Files

- **`rules.v4`** — reference snapshot of `/etc/iptables/rules.v4` on the VPS
  as of 2026-06-23 07:39 UTC. Generated by `iptables-save`. Saved by
  `netfilter-persistent` plugin.
- **`rules.v6`** — IPv6 rules (empty, ipv6 disabled in INPUT)
- **`apply.sh`** — idempotent re-apply script (in case rules.v4 needs restoring)

## Why two layers?

The Xneelo cloud security group filters traffic at the Xneelo edge before it
reaches the VPS. But packets that pass the Xneelo firewall still hit the OS's
iptables INPUT chain. Without explicit ACCEPT rules, even packets from
Cloudflare edge IPs get dropped.

## Applying rules on a fresh VPS

```bash
# Copy rules.v4 to the VPS
scp rules.v4 root@vpn-prod-01:/etc/iptables/rules.v4

# Install netfilter-persistent (if not already)
apt install -y iptables-persistent netfilter-persistent

# Load rules
iptables-restore < /etc/iptables/rules.v4
netfilter-persistent save

# Verify
iptables -L INPUT -n --line-numbers
```

## Audit findings (2026-06-23)

Critical bug: TCP 80 + 443 were missing from INPUT. Fixed by adding rules
9 + 10. Verified from external IP (OC host 192.168.10.77).

See `docs/ISSUES-LOG.md` for full audit + remaining medium/low issues:
- Charon stroke :4502 should be 127.0.0.1 only (currently 0.0.0.0/0)
- 10.99.0.0/24 should be on FORWARD chain (currently on INPUT)
- iptables-nft backend empty + ACCEPT (we use legacy)
